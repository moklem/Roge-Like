# Phase 10 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Project type note:** This is a Godot 4.6 GDScript game (gl_compatibility renderer), not a web app — there is no npm/shadcn/Tailwind toolchain (`components.json` absent, confirmed). The design system is the project's existing hand-rolled **Comic UI Pass** (`scenes/ui/UiStyle.gd`). This contract adapts the standard GSD UI-SPEC template to that system and adds a **Game-Feel/VFX Design Contract** section, since most of Phase 10's requirements are world-space presentation effects (damage numbers, particles, screen shake, progress rings) rather than Control-node UI.

## Spacing Scale

Declared values (must be multiples of 4) — governs **new Control layout** (the Settings sub-panel, any new CardOverlay layout containers):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps |
| sm | 8px | Label-to-slider / label-to-control gap within a settings row |
| md | 16px | Default row separation (matches existing `UiStyle.style_buttons` box-container override of 16) |
| lg | 24px | Section padding, CardOverlay row separation (matches existing `CardsRow` separation of 24) |
| xl | 32px | Gap between distinct settings groups (shake / music / sfx / close) |
| 2xl | 48px | Not used this phase |

**Exceptions (existing, preserved as-is — these are `StyleBoxFlat` visual-style constants inherited from `UiStyle.comic_box()`, not layout-spacing tokens, and are NOT retrofitted to the 4pt scale):**
- `comic_box()` border width: 3px, corner radius: 10px, content margin: 10px all sides, shadow offset: `Vector2(3,3)` normal / `Vector2(1,1)` pressed, shadow size: 4 normal / 1 pressed
- `CardOverlay` existing `OverlayContainer` separation: 10px (unchanged by restyle; only visuals restyled, not layout)
- World-space bar border width: 2px, corner radius: 4px (`style_world_bar`)
- Ink text outline sizes (label 5px, nameplate 6px, level-up label 4px, damage number 4px — new, see Typography)

## Typography

Bangers is a single-weight display font — this project does not vary font-weight; visual hierarchy comes from **size + ink-outline thickness**, not weight. The table below reflects size tiers, all Bangers unless noted.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px | Bangers (single) | 1.3 | CardOverlay desc labels (existing 12px is too small for the restyled comic look — bump to 16px), Settings row labels, HintLabel |
| Label | 20px | Bangers (single) | 1.2 | General UI labels (matches existing `UiStyle.style_label` default 20px) |
| Heading | 26–28px | Bangers (single) | 1.15 | Button text (existing 26px), CardOverlay `TitleLabel` (bump to 28px), Settings panel title |
| Display (world-space, new) | 18–40px continuous | Bangers (single) + 4px black ink outline | 1.0 (single line) | Floating damage numbers (DMG-01) — see Damage Number Scaling below |

**Damage Number Scaling (D-03 — continuous, no crit tiers):**
```
font_size = clamp(18 + damage * 0.55, 18, 40)
```
- Small bolt tick (~5 dmg) → ~21px. Big elemental/upgraded hit (~40 dmg) → ~40px (cap).
- On spawn: punch-scale pop from 1.0 → 1.2 → 1.0 over the first 0.1s (`Tween`, `TRANS_BACK`/`EASE_OUT`).
- Ink outline: 4px black (`font_outline_color = Color(0,0,0,1)`, `outline_size = 4`), matching the established `style_label` convention scaled down for world-space legibility.

## Game-Feel / VFX Design Contract

*(Phase-specific extension — required because 22 of Phase 10's 27 requirements are world-space presentation effects, not Control-node UI. Downstream consumers — planner, executor, ui-checker — treat this section as load-bearing as the sections above.)*

### Element Color Language (shared across every requirement below — build once, reuse everywhere)

| Element | Color | Hex | Used by |
|---------|-------|-----|---------|
| Fire | orange | `#FF8C1A` — `Color(1.0, 0.55, 0.1)` | Damage numbers (D-02), element hit VFX burst (DMG-07), level-up burst when player element = Fire (D-13), evolution burst when player element = Fire (D-14) |
| Ice | light blue | `#7EC8FF` — `Color(0.49, 0.78, 1.0)` | Same list as Fire, Ice variant. Also the existing `ReviveBar` fill color (`Color(0.30,0.65,1.0)`) is visually adjacent but distinct — do not conflate; revive ring keeps its own established blue. |
| Earth | green | `#4CAF6B` — `Color(0.30, 0.69, 0.42)` | Same list as Fire, Ice variant |
| Non-elemental | white | `#FFFFFF` | Screws & Bolts / unelemented weapon damage numbers, generic bursts |

This is a single lookup (`Juice.element_color(element: String) -> Color`) shared by: damage numbers (D-02), element hit VFX (DMG-07), level-up burst (D-13), evolution transform burst (D-14). Do not hand-roll per-effect color logic.

### Damage Numbers (DMG-01)
- Font: Bangers, 4px black ink outline (see Typography).
- Color: element-colored per table above (D-02).
- Size: continuous scale with damage magnitude, 18–40px (D-03, formula above).
- Motion: spawns at hit point, floats upward ~40px over 0.6s while fading out over the final 0.2s; ±8px horizontal jitter to reduce overlap-stacking at swarm volume (Pitfall 5).
- Pooling/aggregation: rapid repeat hits on the same target within a ~100ms window are summed into one number, not stacked (SYS-02, Pitfall 5). Pool size and exact aggregation window are Claude's-discretion per CONTEXT.md — default pool size 24 concurrent numbers, aggregation window 100ms.
- Parented to the persistent `FxLayer`, never to the struck Enemy node (Pitfall 3/4).

### Hit-Flash (DMG-02, D-04)
- Enemy hit-flash: white tint pop, `modulate` flash to `Color(2,2,2,1)`-equivalent (over-bright white) and back over ~0.1s.
- Player hit-flash: red/white alternation, `modulate` flash to `Color(1.0, 0.3, 0.25, 1)` then back over ~0.15s.

### Screen Shake (DMG-03, DMG-08)
- Technique: trauma-accumulator (`trauma` clamped 0–1, decays ~2.5/sec, magnitude derived from `trauma^2`) — never ad hoc offset-and-subtract.
- Intensity tiers (D-08/D-09/D-10/D-11, governs shake ONLY — hit-stop/flash/particles always play regardless of this setting):
  | Setting | Trauma multiplier |
  |---------|-------------------|
  | Off | 0.0× |
  | Low | 0.4× |
  | Normal (default) | 1.0× |
- Default per-client value: **Normal** (D-10). Never synced across peers (D-10) — this is the ONLY per-client, non-networked setting in the whole game.
- Max offset amplitude: ~16px at `trauma = 1.0`; max rotation: ~0.05 rad. Scoped to each peer's own local `Camera2D` only, triggered only by that peer's own damage taken — never by a teammate's hit (Pitfall 5).
- Exact decay-rate/amplitude constants are Claude's-discretion (per CONTEXT.md) — values above are the planning default, tunable against the "subtle & snappy" target (D-06).

### Hit-Stop (DMG-05, D-06)
- Local, per-peer cosmetic float only (`Juice.cosmetic_delta()`) — **never** `Engine.time_scale`, **never** `SceneTree.paused` (hard constraint, Pitfall 1).
- Duration: 60–80ms on a normal enemy kill (target 70ms default); up to ~120ms on Elite/Boss kills for extra weight.
- Read only by presentation code (camera shake decay, sprite flash tweens, particle timestep) — never by movement/AI/cooldown/RPC-dispatch code.

### HP Bar Ghost Chip-Away (DMG-04, D-07)
- Main bar fill snaps instantly to the new HP value (no tween on the primary fill).
- A ghost overlay segment renders where the lost HP was: color `Color(1.0, 0.3, 0.25, 0.85)` (white/red per D-07), positioned from old-value to new-value, drains (shrinks toward the new-value edge, alpha fades to 0) over ~0.4s.
- Applies to both the player's own `HealthBar` (`PlayerHUD`/world-space bar) and Enemy health bars — same treatment, shared helper.

### Death Particle Burst (DMG-06, D-04)
- `CPUParticles2D` one-shot, colored by the dying entity's own color (normal enemy / Elite purple / Boss color — "normal/elite/boss read differently" per D-04).
- Must fire via an RPC carrying `global_position` **before** `queue_free()` (Pitfall 3/4) — never assume the dying node survives to next frame.
- Parented to the persistent `FxLayer`, never to the dying Enemy node.

### Element Hit VFX (DMG-07, D-05)
- Burst-at-impact only, ~0.4s lifetime, element-colored per the shared color table above (fire embers, ice shards, earth chunks). **No lingering ground decals** (explicitly deferred, D-05).
- Depends on the ABIL-01 status-sync fix landing first (per roadmap wave 4 sequencing) — element hit VFX must read the newly-replicated `is_burning`/`is_slowed` flags, not the host-only direct `modulate` write.

### XP Orb Magnetism + Travel-to-Bar (PICK-01/PICK-02, D-15)
- Magnetism: cosmetic-only drift toward the nearest player within ~90px radius, ~260px/s drift speed. Real collection RPC (`_request_collect`) stays untouched — no new networked orb state.
- Travel-to-bar: straight fast dart — ghost orb clone shoots directly from collection point to the XP bar in ~0.3s (`Tween`, ease-in for anticipation or linear — Claude's discretion, default `TRANS_CUBIC`/`EASE_IN`).
- Bar value only increases on arrival (XP bar display value decoupled from the true replicated `xp` field), with a small pulse (scale 1.0→1.1→1.0 over 0.15s) on tick-up. Pulse uses the Accent color (`#FFD640`) as a brief fill-flash.

### Level-Up Burst (PROG-01, D-13)
- Burst effect around the player, element-colored per the shared color table (D-13), triggered on the existing `is_picking_card` diff (Pattern A, zero new RPC).

### Card Overlay Comic Restyle + Pop-In (PROG-02, D-12)
Shared component for both level-up and sub-room weapon-choice presentations — restyle once, both call sites inherit it.

**Restyle (currently zero `UiStyle` calls, confirmed via direct read — fully greenfield):**
- `OverlayBackground` (ColorRect, currently flat `Color(0,0,0,0.55)`): keep as a dim scrim behind the comic panel, unchanged color — this is the modal backdrop, not the panel itself.
- Each card (`Card0`/`Card1`/`Card2`, currently flat `ColorRect` borders `0.35,0.35,0.4` unselected): replace with `UiStyle.comic_box(UiStyle.PAPER)` for unselected, and an **accent-highlighted** variant for the selected card — thicker border glow using the hover-yellow accent color (`#FFD640`) as the border tint, or `UiStyle.comic_box(Color(1.0,0.84,0.25))` (reuses the existing hover stylebox for consistency) to make "currently selected" unmistakable.
- `TitleLabel`: `UiStyle.style_label()` + bump to Heading tier (28px) per Typography table above.
- `Card{N}TypeLabel`/`NameLabel`/`DescLabel`/`HintLabel`: `UiStyle.style_label()` applied (currently unstyled defaults); DescLabel bumped from 12px → 16px (Body tier) for comic-font legibility.
- Icons (`Card{N}Icon`): unchanged — no new icon assets this phase.

**Pop-in animation (PROG-02):**
- On `show_cards()`: `OverlayBackground` alpha fades 0 → 0.55 over 0.15s.
- `OverlayContainer` (or each card individually) scale-pops in: starts at `0.7` scale, animates to `1.0` with a slight overshoot to `1.05` before settling — `Tween`, `TRANS_BACK`, `EASE_OUT`, total duration ~0.25s. Pivot at container/card center.
- Must remain local `CanvasLayer` UI — never `SceneTree.paused` (same discipline as Phase 6's W4, reiterated in Pitfall/Anti-Pattern list).

### Evolution Transform (PROG-03, D-14)
- Sequence: ~0.5s charge-up (glow/shake build-up on the character) → element-colored particle burst (shared color table) + sprite swap to the new stage.
- Total capped at ~1–1.5s, non-blocking, **no camera lock, no input freeze** — for the transforming player or any teammate (hard constraint, explicitly out-of-scope-to-violate per Out of Scope table in REQUIREMENTS.md).
- Screen-space effects (shake/flash) gated `is_multiplayer_authority()` — only the transforming player's own screen shakes/flashes; particle burst + sprite swap visible identically to all peers via the existing `set_evolution_stage` RPC.

### Downed / Revive (COOP-01/02/03, D-18)
- **Downed collapse (COOP-01):** sprite tips 90° (`Tween` rotate, ~0.3s, `EASE_OUT`) + desaturates (modulate lerps toward `Color(0.55,0.55,0.55,1)` over the same 0.3s) + small dust puff (`CPUParticles2D`, brown/grey, ~12 particles, one-shot, at the player's feet).
- **Revive progress ring (COOP-02):** circular ring around the downed player (radius ~28px, 4px stroke, black ink outline consistent with the world-bar style), color `Color(0.30, 0.65, 1.0)` (matches the existing `ReviveBar` fill color). Fills clockwise 0→360° as revive progress goes 0→1. **Must be team-visible** — requires widening `set_revive_progress` from single-target `rpc_id` to a broadcast `.rpc()` (confirmed live gap, RESEARCH.md Pattern B example) since Player nodes are deterministically named (`Player_%d`) and safe to broadcast-target.
- **Revive success (COOP-03):** green sparkle burst (`CPUParticles2D`, `Color(0.4,1.0,0.4)`, one-shot) + ring flashes white then fades + player color snaps back to normal over ~0.2s. Team-visible via the existing `is_downed` true→false diff (Pattern A, zero new RPC).

### Enemy Spawn Telegraph (ABIL-06, D-19)
- Cosmetic only — enemy is active immediately as today, zero authoritative gameplay change (hard constraint).
- ~0.4s materialize effect: fade-in (`modulate.a` 0→1) + an expanding ground ring (neutral white/pale color, since enemy identity/element isn't yet meaningful at spawn), scale pop 0.2 → 1.2 → 1.0 synced with the fade.

### Ability Juice (ABIL-02/03/04/05, D-20)
Shared visual direction: **ghost afterimages + soft glows.**
- Speedster dash (ABIL-02): fading sprite afterimages trailing the dash path, 3–4 copies at decreasing alpha, ~0.3s total fade.
- Tank aura (ABIL-04): expanding soft ring pulse in the aura's element/role color, ~0.4s expand-and-fade.
- Engineer heal (ABIL-03, also satisfies COOP-04 team visibility per D-17): green sparkle rise on the healed player — already structurally team-visible via the existing `_last_health_seen` diff pattern (RESEARCH.md confirms: "health is synced, so every peer sees the burst").
- Drone deploy (ABIL-05): small pop-in burst + brief ring at the deploy point, hooked into `HealDrone._ready()` (already runs identically on every peer).

### Team-Visible Big Hit (COOP-05, D-16)
- Reuses the existing ≥15-damage single-hit threshold (same trigger site as Phase 7's SUSPENSION check in `Player.receive_damage()`) — one shared definition, no new threshold to design.
- New `GameEvents.emit_big_hit(pos)` RPC, mirrors the existing `emit_hud` reliable-broadcast pattern (RESEARCH.md Pattern B) — every peer independently renders a shared visual cue at the hit position (exact visual treatment — e.g. a brief radial pulse or flash ring at the position — is Claude's-discretion within the established "burst/pulse" visual vocabulary already used throughout this contract; reuse the ImpactBurst builder rather than inventing a new one).

### Settings Sub-Panel Layout (D-08/D-09/D-10/D-11)
- Location: new button on `MainMenu.tscn`, below existing `StatusLabel` in the `VBoxContainer` (or as a sibling `Button` in the same container — matches existing menu structure).
- Panel: comic-styled (`UiStyle.comic_box(UiStyle.PAPER)`), opened via the "Settings" button, closed via a "Close" button.
- Contents, top to bottom, using the new spacing scale (32px between groups, 16px row separation, 8px label-to-control gap):
  1. Title label "SETTINGS" (Heading, 28px)
  2. "SCREEN SHAKE" label + 3-state cycle button reading "OFF"/"LOW"/"NORMAL"
  3. "MUSIC VOLUME" label + `HSlider` (0.0–1.0)
  4. "SFX VOLUME" label + `HSlider` (0.0–1.0)
  5. "Close" button
- All settings styled via the existing `UiStyle.style_buttons()`/`style_labels()` sweep (no bespoke styling needed — these are native `Button`/`Label`/`HSlider` nodes under the panel root).
- Persistence: in-memory per launch, no config-file persistence required this phase (Claude's discretion, confirmed no user requirement).
- **Implementation prerequisite (RESEARCH.md Pitfall 7):** `Sfx.gd`/`Music.gd` currently hard-code bus `"Master"` — new `"Music"`/`"SFX"` audio buses must be created before the sliders can function; this is a planning/build dependency, not a UI-visual concern, but the planner must sequence it before the sliders are wired.
