<objective>
Build the foundational, purely-additive juice infrastructure that every later wave consumes: a `Juice` execution autoload (trauma-based screen shake, local cosmetic hit-stop, pooled/aggregated floating damage numbers, hit-flash, parametrized CPUParticles2D bursts, shared spawn+cleanup with backstop timers, and the one shared element-color lookup), a `Settings` client-only autoload (shake intensity + music/sfx volume state), the parametrized `scenes/vfx/` builders, and the persistent `FxLayer` container under `Game.tscn`. Registers both autoloads in `project.godot`.

No gameplay file is edited in this plan — the infrastructure must exist and be verifiable in isolation before any consuming effect is built (SYS-01/02/03). Implements the hard constraints once, centrally: CPUParticles2D only, hit-stop as a local per-peer cosmetic float (never engine-global time scaling), transient VFX parented to `FxLayer`, and pooled/capped damage numbers + trauma shake shared by all later waves.

Purpose: One correct, reusable juice engine so ~10 later effect types do not each hand-roll particle construction, shake decay, or cleanup (the exact duplication risk RESEARCH flags between the existing `_spawn_heal_particles`/`_spawn_driver_particles`).
Output: `Juice` + `Settings` autoloads, `scenes/vfx/` builders, `FxLayer` node, `project.godot` registration.
</objective>
