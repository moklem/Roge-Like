<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → host (XpOrb._request_collect RPC) | Client claims it collected an orb; host must validate before granting XP |
| host → owning peer (receive_xp RPC) | Host pushes authoritative XP amount to the owning peer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Spoofing | XpOrb._request_collect | mitigate | Host identifies collector via `multiplayer.get_remote_sender_id()` (not a client-supplied peer_id field); only the touching peer sends the RPC (existing `_on_body_entered` peer guard) |
| T-06-02 | Tampering | XP amount | mitigate | XP amount (15) is hardcoded on the host inside `_request_collect`; clients cannot supply the amount |
| T-06-03 | Elevation | double-grant via race | mitigate | Existing `_collected` guard set before grant ensures a single grant per orb (W5) |
| T-06-04 | Tampering | client-incremented XP | mitigate | `xp` is only incremented inside `receive_xp`, which runs on the owning peer after a host-initiated rpc_id; clients never increment xp from `_on_body_entered` (P8) |
</threat_model>
