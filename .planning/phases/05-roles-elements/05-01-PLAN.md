## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| owning client → host | Ability input (Space) is read only on the owning peer; effects must route to host |
| host → owning peer | receive_heal / set_evolution_stage RPCs sent host→peer via rpc_id |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Elevation (authority abuse) | Player._apply_role_stats sets HP/SPEED | accept | Stats set in `_ready()` on each peer's own node from authoritative Lobby.players; no client can grant another player higher stats — each node applies its own role only |
| T-05-02 | Spoofing (RPC) | receive_heal / set_evolution_stage | mitigate | Both are `@rpc("any_peer", "call_remote")`; only invoked by host (Plan 03/05 call sites guard `multiplayer.is_server()`); owning peer applies to own health/stage which then replicates — a spoofed client call only heals itself (no cross-peer effect) |
| T-05-03 | Information disclosure / desync | shield_active / dash_invincible / evolution_stage | mitigate | Added to Player MultiplayerSynchronizer replication config (replication_mode=2) so all peers observe the same flag state at 20 Hz; no peer renders a stale ability visual |
| T-05-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase — pure Godot 4.6 GDScript (RESEARCH Package Legitimacy Audit: not applicable) |
</threat_model>
