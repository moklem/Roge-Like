```gdscript
## HLTH-05: Called by Player.gd each frame the reviver holds E near a downed player
## Accumulates time; when >= REVIVE_DURATION calls revive on the target
@rpc("any_peer", "call_remote", "reliable")
func attempt_revive(reviver_id: int, target_id: int) -> void:
	if not multiplayer.is_server():
		return
	# Validate: target must be downed, reviver must be close enough and not downed
	var reviver: Node = null
	var target: Node = null
	for p in get_tree().get_nodes_in_group("players"):
		if p.peer_id == reviver_id:
			reviver = p
		if p.peer_id == target_id:
			target = p
	if reviver == null or target == null:
		return
	if not target.is_downed:
		# Target already revived (race condition) — reset progress and ignore
		_revive_progress.erase(target_id)
		return
	if reviver.is_downed:
		return  # cannot revive while downed
	if reviver.global_position.distance_to(target.global_position) > REVIVE_PROXIMITY:
		# Reviver walked away — Pitfall 6: reset progress
		_revive_progress.erase(target_id)
		_update_revive_bar(target_id, 0.0)
		return
	# Accumulate revive progress
	var dt: float = get_physics_process_delta_time()
	var progress: float = _revive_progress.get(target_id, 0.0) + dt
	_revive_progress[target_id] = progress
	# HLTH-06: Update progress bar on the target player node (visible to all via _process)
	var pct: float = minf(progress / REVIVE_DURATION, 1.0)
	_update_revive_bar(target_id, pct)
	if progress >= REVIVE_DURATION:
		# Revive complete
		_revive_progress.erase(target_id)
		# Call revive on the owning peer — target.revive() is a local method (not RPC)
		# We need to call it via RPC to reach the owning peer
		target.revive.rpc_id(target_id)  # revive() is NOT an RPC — use set_revive via signal
		# SAFER: just call target.revive() directly — this runs on host and on the target's node
		# Since Player nodes have set_multiplayer_authority(peer_id), calling .revive() on host
		# modifies the host-side node. The health and is_downed properties will sync via
		# MultiplayerSynchronizer back to the owning peer. This is Option 1 from Pitfall 3.
		# revive() sets health = MAX_HP/2 and is_downed = false on the host-side instance;
		# those sync outward via the existing MultiplayerSynchronizer.
		# This requires that health and is_downed in Player.tscn have their Synchronizer
		# authority set to ALLOW_ALL (replication_mode = 2 = ALWAYS), which allows host to write.
		target.revive()

## Update the ReviveBar on the target player — visible to all peers from synced position
func _update_revive_bar(target_id: int, progress: float) -> void:
	for p in get_tree().get_nodes_in_group("players"):
		if p.peer_id == target_id:
			if p.has_method("set_revive_progress"):
				p.set_revive_progress(progress)
			break
```

IMPORTANT NOTE on revive(): The call `target.revive()` runs on host's copy of the player node. Because Player.tscn's MultiplayerSynchronizer has `replication_mode = 2` (ALWAYS) for health and is_downed, and the synchronizer is set to the owning peer's authority, the host's direct write to `health` and `is_downed` on a non-host player node will NOT propagate outward automatically unless the synchronizer authority is the host.

The simpler and more correct approach for this project: use rpc_id to the owning peer:
```gdscript
# Replace target.revive() with:
target.receive_revive.rpc_id(target_id)
```
And add to Player.gd:
```gdscript
@rpc("authority", "call_remote", "reliable")
func receive_revive() -> void:
    revive()
```

But Player.gd cannot have `@rpc("authority")` calls FROM the host to a client with "call_remote" (authority = node's multiplayer authority = the owning peer; host is NOT the authority, so host cannot use `@rpc("authority")` to send to the client).

CORRECT APPROACH: Use `@rpc("any_peer", "call_remote", "reliable")` for receive_revive on Player.gd so the host (any_peer) can call it on the target's peer:
```gdscript
@rpc("any_peer", "call_remote", "reliable")
func receive_revive() -> void:
    revive()
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Host (fire request) | Client sends request_fire with direction; host ignores client-supplied position |
| Client → Host (revive request) | Client sends attempt_revive; host validates proximity and downed state |
| Host → All (game over) | Host broadcasts via call_local RPC; all peers change scene |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-15 | Spoofing | request_fire position exploit | mitigate | Host ignores `_client_pos` parameter; uses server-authoritative `player_node.global_position` from the MultiplayerSynchronizer-replicated node. Client cannot teleport bullets. |
| T-03-16 | Tampering | attempt_revive validation bypass | mitigate | Host validates: target.is_downed must be true, reviver must be within REVIVE_PROXIMITY, reviver must not be downed. Client cannot force a revive through invalid state. |
| T-03-17 | Spoofing | _broadcast_game_over unauthorized trigger | mitigate | `@rpc("authority", ...)` — only GameState's authority (host, peer 1) can call this RPC. Clients cannot send it. |
| T-03-18 | Elevation of Privilege | Enemy spawning by client | mitigate | _spawn_enemies() guarded by `multiplayer.is_server()` in _ready(); EnemySpawner.spawn() is called only inside that guard. No client-callable RPC for enemy spawn. |
| T-03-19 | Tampering | Revive progress manipulation | mitigate | _revive_progress dictionary lives only on host; clients send frame-by-frame attempt_revive RPC; host is single source of truth for progress accumulation. |
| T-03-20 | Denial of Service | receive_revive spam | accept | revive() sets is_downed=false; subsequent calls are no-ops (reviving an already-alive player has no negative effect). Low risk. |
