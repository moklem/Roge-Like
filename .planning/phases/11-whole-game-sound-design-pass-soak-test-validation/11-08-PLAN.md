  <what-built>
    Plans 11-02..11-07 wired the whole-game sound layer: extended voice pool with a reserved priority subset (Sfx.gd), the central GameEvents/hud_event listener with onset debounce, the Music sting layer, and cue calls across weapons, abilities, elements, pickups, UI, transitions, enemies, and boss. This test confirms none of that (nor the Phase 10 juice it sits on) leaks nodes over a full loop, and that priority stingers keep playing.
  </what-built>
  <how-to-verify>
Run from the Godot editor so the Debugger Monitors and Remote scene tree are available.

PART A — HOST role:
1. Launch one instance, Host a session, start a solo run (host is playable solo).
2. Open Debugger → Monitors. Record the baseline once combat begins: Object Count, Node Count, Orphan Node Count.
3. Play (or idle-with-combat) continuously for ~15 minutes through at least one full loop (Room 1 → 2 → 3/boss → next loop). Deliberately trigger, repeatedly: enemy kills, elite kills, taking damage, level-ups, an evolution, a downed+revive, and (if reachable) the boss phase-changes and boss death.
4. Every ~5 minutes, and again at the end, record Object/Node/Orphan counts at a comparable moment (similar on-screen enemy count).
5. Open Debugger → Remote scene tree and inspect the persistent `FxLayer` (under Game): confirm it is NOT accumulating hundreds of leftover ColorRect/CPUParticles2D/Label/Tween children between bursts — it should return near baseline when the screen is quiet.
6. Watch the Output/Debugger for errors (nil access, RPC signature errors, "node not found").

PART B — CLIENT role:
7. Repeat steps 1-6 with TWO instances: one host, one client (same machine is acceptable, or a second machine). Focus the monitors on the CLIENT instance. This catches per-peer leaks that only appear on the non-authority side (e.g. spawner-propagated `_exit_tree` death VFX, call_local cue paths).

STINGER-DROP check (both roles): during a busy fight (many routine hit/shoot sounds), deliberately trigger a priority event (evolution, downed, or big-hit/level-up). Confirm the priority cue is still heard (if its file exists) — it must not be silently stolen by routine sounds. If the file is missing, confirm no error and that routine audio keeps working (no wedged/stuck voices).
  </how-to-verify>
