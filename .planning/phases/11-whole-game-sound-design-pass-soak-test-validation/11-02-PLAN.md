```gdscript
## Reserved-voice playback for must-hear stingers. If every reserved voice is busy,
## steal the routine round-robin slot from the shared pool (D-04) rather than dropping.
func _play_priority(stream: AudioStream, volume_db: float, pitch_min: float, pitch_max: float) -> void:
    if stream == null:
        return
    var p: AudioStreamPlayer = null
    for cand in _priority_players:
        if not cand.playing:
            p = cand
            break
    if p == null and not _priority_players.is_empty():
        # All reserved voices busy — round-robin a reserved slot AND, per D-04, prefer stealing
        # from the shared pool so a second simultaneous priority cue is never dropped.
        p = _priority_players[_priority_next]
        _priority_next = (_priority_next + 1) % _priority_players.size()
    if p == null:
        # No reserved voices configured — fall back to the shared pool.
        _play(stream, volume_db, pitch_min, pitch_max)
        return
    p.stream = stream
    p.volume_db = volume_db
    p.pitch_scale = randf_range(pitch_min, pitch_max)
    p.play()
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GameEvents broadcast → Sfx listener | The central listener reacts to `hud_event`/`player_downed`/`player_revived`/`big_hit`. All four are emitted by `@rpc("authority", "call_local", ...)` handlers in GameEvents.gd — host-authoritative, never client-originated. |
| Voice pool resource | Finite `AudioStreamPlayer` array; the priority overflow logic could in theory be exercised to force voice churn. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-02 | D (DoS: voice-pool exhaustion) | Sfx priority pool + central listener | mitigate | The listener only reacts to `GameEvents` signals that are emitted from `@rpc("authority", ...)` handlers — a malicious client cannot emit them directly. The pool is a fixed 20 `AudioStreamPlayer`s; `_play`/`_play_priority` only ever reuse existing slots (never allocate), so audio spam cannot grow memory or node count. `HUD_ECHO_DEBOUNCE` (1.2s) further throttles the highest-frequency echo path. |
| T-11-03 | E (Elevation) / T (Tampering) | cue name → file path | accept | All cue paths are hardcoded `const`/literal `res://assets/audio/...` strings loaded via `_try_load`/`ResourceLoader.exists`. No cue path is ever built from peer-supplied data, so there is no path-traversal or arbitrary-file-load vector. |
| T-11-04 | S (Spoofing) | player_downed(id)/big_hit(pos) payloads | accept | The listener ignores the payloads for audio purposes (`downed()`/`big_hit()` take no args). A spoofed id/pos could at most trigger a sound already gated by the host-authoritative emit; no state is mutated. |

No high-severity findings. Only new behavior is presentation audio reacting to existing host-authoritative broadcasts on a bounded, pre-allocated voice pool.
